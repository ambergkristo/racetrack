# 27.03 Sprint Integration Statute

## Eesmärk
See dokument määrab ametliku post-M3 faasipõhise arendus-, integratsiooni-, gate'i- ja merge-korra 27.03 sprinti jaoks.

- M2.5 ja M3 on valmis ning lukustatud `main` peal.
- Iga järgmine faas peab edenema kontrollitud integratsioonina, mitte lõpu-merge'ina.
- See seadus kehtib nii inimestele kui ka tulevastele Codex sessioonidele.

## Põhireegel
Post-M3 faase ei tehta mudeliga "kõik lõpetavad kõik ära" ega "üks hiigelmerge lõpus".

Iga faasi ametlik töövoog on:
1. aktiivsed arendajad töötavad omaenda faasi dev branchides
2. valmis töö kogutakse faasi integration branchi
3. integration branchil tehakse nõutud gate, parity, truth ja regression kontroll
4. ainult PASS integration branch võib liikuda `main`-i
5. järgmine faas algab alati värske `main` pealt

## Phase integration workflow
Iga faasi ametlik integratsioonikäik on järgmine:

### 1. Dev branches
- Iga aktiivne arendaja töötab ainult oma selle faasi branchis.
- Arendaja branch ei lähe otse `main`-i.

### 2. Phase integration branch
- Kui aktiivsete arendajate selle faasi töö on valmis või valmis integratsiooniks, luuakse sellele faasile eraldi integration branch.
- Integration branch koondab ainult selle faasi jaoks nõutud muudatused.

### 3. Gate / validation
- Gate käib integration branchil, mitte üksikutes dev branchides.
- Gate peab katma vähemalt truth, parity, regression, smoke või demo readiness kontrolli vastavalt faasi eesmärgile.
- Kui gate FAIL, ei tohi `main` merge'i teha.

### 4. Merge to main
- `main` merge on lubatud ainult siis, kui faasi integration branch on PASS.
- Merge peab põhinema integration branchi kontrollitud seisul.

### 5. Next phase
- Järgmine faas algab alati värskeima `main` pealt.
- Stale branchist järgmise faasi tööd alustada ei tohi.

## Phase-by-phase order
### Phase 2 - P0 UX correction
Aktiivsed arendajad:
- Dev A
- Dev B
- Dev C
- Dev D alles pärast esimest kasutatavat passi

Kord:
- Dev A, Dev B ja Dev C alustavad paralleelselt oma Phase 2 branchides.
- Dev D ei alusta regressiooni kohe.
- Dev D alustab alles siis, kui A/B/C poolt on olemas esimene kasutatav UI või API pass.
- Seejärel luuakse `integration/phase-2-ux-correction`.
- Integration branchi viiakse A/B/C muudatused.
- Dev D teeb regression/gate kontrolli integration branchil.
- Ainult PASS järel tohib Phase 2 minna `main`-i.

### Phase 3 - Front desk workflow
Aktiivsed arendajad:
- Dev A
- Dev B
- Dev D
- Dev C ainult siis, kui on vaja ühilduvuse sisendit

Kord:
- Dev A ja Dev B alustavad paralleelselt.
- Dev D alustab pärast seda, kui A/B workflow muudatused on integration-ready.
- Seejärel luuakse `integration/phase-3-front-desk-workflow`.
- Integration branchi viiakse A/B muudatused.
- Dev D teeb workflow regression kontrolli integration branchil.
- Ainult PASS järel tohib Phase 3 minna `main`-i.

### Phase 4 - Race control clarity
Aktiivsed arendajad:
- Dev A
- Dev B
- Dev C
- Dev D pärast päris muudatuste olemasolu

Kord:
- Dev A ja Dev B alustavad paralleelselt.
- Dev C alustab pärast või Dev B lõpuosas, et public state language peegeldaks race-control truth'i.
- Seejärel luuakse `integration/phase-4-race-control-clarity`.
- Integration branchi viiakse A/B/C muudatused.
- Dev D teeb regression/gate kontrolli integration branchil.
- Ainult PASS järel tohib Phase 4 minna `main`-i.

### Phase 5 - Public display polish
Aktiivsed arendajad:
- Dev C lead
- Dev B support
- Dev D regression
- Dev A ainult siis, kui backend support on selgelt vajalik

Kord:
- Dev C alustab esimesena.
- Dev B joondab semantika.
- Dev D alustab alles pärast esimest kasutatavat public passi.
- Vajadusel lisatakse Dev A backend support.
- Seejärel luuakse `integration/phase-5-public-display-polish`.
- Integration branchi viiakse C/B ja vajadusel A muudatused.
- Dev D teeb display regression kontrolli integration branchil.
- Ainult PASS järel tohib Phase 5 minna `main`-i.

### Phase 6 - Demo readiness
Aktiivsed arendajad:
- kõik arendajad

Kord:
- Kõik arendajad töötavad omaenda Phase 6 branchides.
- Seejärel luuakse `integration/phase-6-demo-readiness`.
- Integration branchi viiakse kõik nõutud demo-readiness muudatused.
- Integration branchil tehakse final smoke, demo ja fallback kontrollid.
- Ainult PASS järel tohib Phase 6 minna `main`-i.

## Integration Reminder
See jaotis on repo-seadus post-M3 faaside jaoks.

- Arendaja branch ei lähe nende post-M3 faaside jooksul KUNAGI otse `main`-i.
- Enne merge'i `main`-i peab Codex kontrollima:
1. kas kõik selle faasi aktiivsed dev branchid on jõudnud faasi integration branchi
2. kas nõutud faasi gate või regression on käivitatud
3. kas integration branch on PASS
- Kui mõni neist tingimustest ei ole täidetud, peab Codex peatuma ja ütlema:
`Phase integration is required before merge to main.`

## Gate before main
Enne `main` merge'i peab igal faasil olema järgmine miinimum:

- Phase 2: truth, usability ja regression gate integration branchil
- Phase 3: workflow regression gate integration branchil
- Phase 4: race-control clarity, public language mirror ja regression gate integration branchil
- Phase 5: public display readability, semantics alignment ja display regression gate integration branchil
- Phase 6: final smoke, demo flow ja fallback gate integration branchil

PASS või FAIL peab olema sõnaselgelt kirjas. Vaikiv või oletuslik valmisolek ei kehti.

## No scope drift
Keelatud on:

- ära ava M3 uuesti
- ära disaini süsteemi nullist ümber
- ära muuda lifecycle truth'i
- ära lõhu OFF/ON feature-flag käitumist
- eelista minimaalseid, kõrge väärtusega muudatusi
- iga faas peab lõppema tõendi ja PASS/FAIL tulemusega
- `main` jääb faaside vahel ainsaks tõeallikaks

## Evidence rules
Iga faasi merge-eelne tõend peab sisaldama vähemalt:

- viide aktiivsetele dev branchidele, mis integration branchi maandusid
- kirje selle kohta, millal integration branch loodi
- gate või regression tulemused
- vajadusel screenshot, route verification, smoke log või parity märge
- selge PASS või FAIL otsus

Ilma tõendita faasi ei loeta integratsioonivalmis.

## Final merge rule
Lõplik merge-reegel on järgmine:

- ainult faasi integration branch võib olla `main` merge alus
- ainult PASS integration branch võib minna `main`-i
- pärast merge'i järgmine faas algab värskelt `main` pealt
- `main` on faaside vahel ametlik source of truth
