# 27.03 Sprint Overview

## Sprint purpose
27.03 sprint on post-M3 kasutatavuse, workflow ja demo-valmiduse sprint. Eesmärk ei ole lisada uut toodet ega avada uuesti M3 scope'i, vaid teha olemasolev `main` peal lukustatud süsteem operaatorile, publikule ja demole kohe arusaadavaks.

## Locked baseline note
- M2.5 on tehtud.
- M3 on tehtud.
- `main` on tõe baasliin.
- `main` sisaldab juba lifecycle truth'i, feature-flag matrix'it, persistence'it ja manual assignment'it.
- See sprint on ainult post-M3 UX, workflow ja demo töö.

## Phase-by-phase summary
- Phase 2: P0 UX correction sprint. Kõik põhivaated peavad muutuma ühe pilguga arusaadavaks ja 1080p peal kasutatavaks.
- Phase 3: Front desk workflow completion. `/front-desk` peab toetama päris queue + current + next operatsioonivoogu, mitte ainult CRUD-i.
- Phase 4: Race control clarity sprint. Safety officer peab nägema kohe, kas süsteem on STAGING, RUNNING, FINISHED või LOCKED ning milline tegevus on lubatud.
- Phase 5: Public display polish sprint. Public route'id peavad andma vajaliku info 2–3 sekundiga, mitte jätma dashboard-demo muljet.
- Phase 6: Demo readiness sprint. Lukk panna 5–7 minuti demo, kuivjooksud, fallback ja tõenduspõhine readiness.

## Dev-by-dev responsibilities table
| Dev | Phase 2 | Phase 3 | Phase 4 | Phase 5 | Phase 6 |
| --- | --- | --- | --- | --- | --- |
| Dev A | Canonical snapshot + derived fields + state mapping truth | Queue ordering truth + guards | FINISHED/LOCKED eristus backendis | Ainult vajadusel backend tugi | Backend truth + viimased selgitavad parandused |
| Dev B | Staff route usability | Front desk workflow lead | Race control clarity lead | Semantika kooskõla kontroll staff/public vahel | Front desk ja operator flow demo readiness |
| Dev C | Public route usability lead | Mitte aktiivne, ainult vajadusel ühilduvusmärkus | Public state language mirror | Public display polish lead | Public demo coherence |
| Dev D | UX truth regression coverage | Queue/workflow regression coverage | Transition regression coverage | Fullscreen smoke + readability proof | Release checklist, smoke, fallback |

## Prompt order by phase
- Phase 2: Dev A, Dev B ja Dev C alustavad paralleelselt. Dev D alustab alles siis, kui A/B/C on esimese kasutatava UI või API muudatuse pushinud või merge'inud.
- Phase 3: Dev A ja Dev B alustavad paralleelselt. Dev D alustab pärast seda, kui A/B front-desk workflow muudatused on kättesaadavad. Dev C ei ole aktiivne, välja arvatud väike ühilduvusmärkus.
- Phase 4: Dev A ja Dev B alustavad paralleelselt. Dev C alustab pärast või B töö lõppfaasis, et public state language peegeldaks race-control truth'i. Dev D alustab pärast A/B/C muudatuste kättesaadavust.
- Phase 5: Dev C juhib. Dev B kontrollib staff/public semantika kooskõla. Dev D alustab pärast C esimest kasutatavat public-display passi. Dev A ei ole aktiivne, välja arvatud explicit backend support.
- Phase 6: Kõik panustavad. Dev D omab release checklist'i, smoke'i ja fallback'i. Dev C omab public demo koherentsi. Dev B omab front-desk/operator readiness'i. Dev A toetab backend truth'i ja viimaseid täppiskorrektuure.

## Dependency rules
- M3 ei avata uuesti. Kõik tööd lähtuvad `main` peal olevast lukustatud käitumisest.
- Dev D ei tohi alustada regressioonide ja smoke'idega enne, kui tal on päriselt olemas A/B/C poolt pushitud või merge'itud kasutatav vaheolek.
- Dev C ei tohi Phase 4 keelelist peegeldust teha oletuste pealt; tal peab olema B race-control sõnavara ja A state truth käes.
- Dev B Phase 3 front-desk workflow ei tohi mööda minna A queue truth'ist; queue/current/next allikas peab jääma backend truth'i külge.
- Phase 6 tõend peab tuginema eelmiste faaside valmisolekule, mitte lootusele.

## No scope drift rules
- Ära ava M3 uuesti.
- Ära disaini süsteemi nullist ümber.
- Ära muuda lifecycle truth'i.
- Ära lõhu OFF/ON flag käitumist.
- Eelista minimaalseid, kõrge väärtusega UX ja workflow muudatusi.
- Iga faas peab lõppema tõendiga, mitte arvamusega.

## Evidence expectations
- Iga aktiivne töö peab andma vähemalt ühe screenshot'i.
- Iga aktiivne workflow või truth-muudatus peab andma testitulemuse.
- Iga route'i puudutav töö peab andma route verification märkme.
- Iga faasifail peab lõppema PASS/FAIL märkega töötegija enda tõendikomplektis.
