# 27.03 Sprint Overview

## Sprint purpose
27.03 sprint on post-M3 kasutatavuse, workflow ja demo-valmiduse sprint.

- Eesmärk ei ole avada uut toodet ega M3 scope'i.
- Eesmärk on teha `main` peal lukustatud süsteem operaatorile, publikule ja demole kiiresti arusaadavaks.

## Locked baseline note
- M2.5 on valmis.
- M3 on valmis.
- `main` on lukustatud baasliin.
- `main` sisaldab lifecycle truth'i, feature-flag parity't, persistence'it ja manual assignment'it.
- Kõik 27.03 sprinti faasid lähtuvad sellest baasliinist.

## Integration workflow rules
Post-M3 faaside ametlik kord on järgmine:

1. aktiivsed arendajad töötavad omaenda faasi dev branchides
2. faasi töö koondatakse eraldi phase integration branchi
3. gate, truth, parity, regression või smoke kontroll tehakse integration branchil
4. ainult PASS integration branch läheb `main`-i
5. järgmine faas algab värskeima `main` pealt

Keelatud on:
- kõik lõpetavad kõik ära ja siis tehakse üks suur lõpu-merge
- arendaja branchi otsene merge `main`-i
- järgmise faasi alustamine stale branchi pealt

## Phase-by-phase summary
- Phase 2: P0 UX correction. Kõik põhivaated peavad muutuma ühe pilguga arusaadavaks ja 1080p peal kasutatavaks.
- Phase 3: Front desk workflow. `/front-desk` peab toetama päris queue + current + next töövoogu, mitte ainult CRUD-i.
- Phase 4: Race control clarity. Safety officer peab nägema kohe, kas süsteem on STAGING, RUNNING, FINISHED või LOCKED ning milline tegevus on lubatud.
- Phase 5: Public display polish. Public route'id peavad andma vajaliku info 2-3 sekundiga, mitte jätma dashboard-demo muljet.
- Phase 6: Demo readiness. Sprint lõpeb 5-7 minuti demo, kuivjooksude, fallback'i ja tõenduspõhise valmisolekuga.

## Phase integration order
### Phase 2
- Dev A, Dev B ja Dev C alustavad paralleelselt.
- Dev D ei alusta kohe.
- Dev D alustab alles siis, kui A/B/C esimene kasutatav pass on olemas.
- Seejärel luuakse `integration/phase-2-ux-correction`.
- A/B/C muudatused integreeritakse sinna.
- Dev D teeb regression/gate kontrolli integration branchil.
- Ainult PASS järel tohib merge'ida `main`-i.

### Phase 3
- Dev A ja Dev B alustavad paralleelselt.
- Dev D alustab alles siis, kui A/B workflow muudatused on olemas.
- Dev C ei ole aktiivne, välja arvatud vajaduspõhine compatibility input.
- Seejärel luuakse `integration/phase-3-front-desk-workflow`.
- A/B muudatused integreeritakse sinna.
- Dev D teeb workflow regression kontrolli integration branchil.
- Ainult PASS järel tohib merge'ida `main`-i.

### Phase 4
- Dev A ja Dev B alustavad paralleelselt.
- Dev C alustab pärast või Dev B lõpuosas, et public state language peegeldaks race-control truth'i.
- Dev D alustab alles pärast päris muudatuste olemasolu.
- Seejärel luuakse `integration/phase-4-race-control-clarity`.
- A/B/C muudatused integreeritakse sinna.
- Dev D teeb regression/gate kontrolli integration branchil.
- Ainult PASS järel tohib merge'ida `main`-i.

### Phase 5
- Dev C alustab esimesena ja juhib faasi.
- Dev B toetab semantika joondust.
- Dev D alustab alles pärast esimest kasutatavat public passi.
- Dev A liitub ainult siis, kui explicit backend support on vajalik.
- Seejärel luuakse `integration/phase-5-public-display-polish`.
- C/B ja vajadusel A muudatused integreeritakse sinna.
- Dev D teeb display regression kontrolli integration branchil.
- Ainult PASS järel tohib merge'ida `main`-i.

### Phase 6
- Kõik arendajad töötavad oma Phase 6 branchides.
- Seejärel luuakse `integration/phase-6-demo-readiness`.
- Kõik vajalikud demo-readiness muudatused integreeritakse sinna.
- Seal tehakse final smoke, demo ja fallback kontrollid.
- Ainult PASS järel tohib merge'ida `main`-i.

## Dependency rules
- M3 ei avata uuesti.
- Dev D ei tohi alustada regressiooni ega smoke'i enne, kui tal on reaalselt testitav vaheolek.
- Dev C ei tohi Phase 4 state language mirror'it teha oletuste pealt; tal peab olema Dev B sõnavara ja Dev A state truth käes.
- Dev B Phase 3 workflow ei tohi mööda minna Dev A queue truth'ist; queue/current/next allikas peab jääma backend truth'i külge.
- Phase 6 tõend peab tuginema eelnevate faaside PASS tulemustele, mitte lootusele.

## Integration Reminder
See on ametlik repo meeldetuletus post-M3 faasideks.

- Arendaja branch ei lähe KUNAGI otse `main`-i.
- Enne merge'i `main`-i peab Codex kontrollima:
1. kas kõik aktiivsed dev branchid on jõudnud faasi integration branchi
2. kas nõutud faasi gate või regression on käivitatud
3. kas integration branch on PASS
- Kui mõni tingimus ei ole täidetud, peab Codex peatuma ja ütlema:
`Phase integration is required before merge to main.`

## No scope drift
- ära ava M3 uuesti
- ära disaini süsteemi nullist ümber
- ära muuda lifecycle truth'i
- ära lõhu OFF/ON feature-flag käitumist
- eelista minimaalseid, kõrge väärtusega muudatusi
- iga faas peab lõppema tõendi ja PASS/FAIL tulemusega
- `main` jääb faaside vahel ainsaks tõeallikaks

## Evidence expectations
- Iga aktiivne töö peab andma vähemalt ühe screenshot'i või route verification'i.
- Iga aktiivne workflow või truth-muudatus peab andma testitulemuse.
- Iga faas peab lõppema selge PASS/FAIL märkega.
- Iga faasi merge-eelne otsus peab olema seotud vastava integration branchiga.
